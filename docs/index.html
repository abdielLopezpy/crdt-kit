What are CRDTs?

Conflict-Free Replicated Data Types are data structures that can be replicated across multiple devices, updated independently and concurrently, and merged automatically without conflicts.

Unlike traditional databases that require a central server to coordinate writes, CRDTs guarantee that all replicas will converge to the same state regardless of the order in which updates are received. This guarantee is mathematical, not dependent on network conditions.

This makes them ideal for offline-first apps, peer-to-peer systems, IoT networks, and any scenario where devices need to work independently and sync later.

Features

no_std / Embedded

Runs on bare metal, Raspberry Pi, ESP32. All types work with #![no_std] + alloc. No heap allocator required for core operations.

Delta Sync

Only send what changed since the last sync. DeltaCrdt trait for GCounter, PNCounter, and ORSet. Minimizes bandwidth on LoRa, BLE, and constrained networks.

WASM

First-class WebAssembly bindings via wasm-bindgen. Use the same CRDTs in browser, Deno, Node.js, and Rust backend. GCounter, PNCounter, LWWRegister, GSet, ORSet, TextCrdt.

Schema Migrations

Transparent, lazy migrations on read. #[crdt_schema] + #[migration] proc macros. Linear chain v1 → v2 → v3. Deterministic — two devices migrating the same data produce identical results.

Persistent Storage

Three backends, one trait surface: SQLite (bundled), redb (pure Rust, no C deps), and in-memory. Event sourcing, snapshots, and compaction built in.

Code Generation

Define entities in TOML, run crdt generate, get a complete persistence layer: versioned structs, migration functions, repository traits, event sourcing, delta sync.

Serde

Serialize / Deserialize for all 9 CRDT types via feature flag. Works with JSON, MessagePack, Postcard, CBOR — any serde-compatible format.

Event Sourcing

Full event log with append, replay, snapshots, and compaction. EventStore trait implemented by all backends. Configurable snapshot policies.

Developer Tools

CLI with status, inspect, compact, export, generate, dev-ui. Embedded dark-themed web panel for visual database inspection.

9 CRDT Types

From simple counters to collaborative text editing. Every type is Send + Sync, serde-ready, and mathematically convergent.

Counters

GCounter

DeltaCrdt

Grow-only counter. Each node maintains its own count; the total is the sum of all nodes. Monotonically increasing.

Page views, IoT sensor events, download counts, heartbeats 
let mut c = GCounter::new("node-a");
c.increment_by(42);
// c.value() == 42

PNCounter

DeltaCrdt

Positive-negative counter. Supports both increment and decrement operations. Internal pair of GCounters.

Inventory stock, likes/dislikes, seat reservations, voting 
let mut c = PNCounter::new("node-a");
c.increment();
c.decrement();
// c.value() == 0

Registers

LWWRegister

State

Last-Writer-Wins register. On concurrent writes, the one with the highest timestamp wins. Uses Hybrid Logical Clock.

User profile fields, GPS location, config settings, status 
let mut r = LWWRegister::new("node-a", "initial".into());
r.set("updated".into());
// r.value() == "updated"

MVRegister

State

Multi-Value register. Preserves all concurrent writes instead of silently discarding. Application resolves conflicts.

Version tracking, audit trails, conflict visualization 
let mut r = MVRegister::new("node-a", "v1".into());
// After concurrent writes: r.values() == ["v1", "v2"]

Sets

GSet

State

Grow-only set. Elements can be added but never removed. Union on merge.

Seen message IDs, tags, audit logs

TwoPSet

State

Two-Phase set. Add and remove, but once removed an element cannot be re-added.

Blocklists, revoked tokens, bans

ORSet

DeltaCrdt

Observed-Remove set. Add and remove elements freely. Concurrent add+remove: add wins.

Shopping carts, chat members, todo lists

Sequences

Rga

State

Replicated Growable Array. Ordered sequence with insert, remove, and move. Position-independent identifiers.

Playlists, kanban boards, ordered lists, timelines

TextCrdt

State

Collaborative text editing. insert_str(), remove_range(), fork(). Built on RGA with string optimizations.

Google Docs-style editing, shared notes, wikis

Traits

Crdt

trait

Core merge semantics. merge(&other) is commutative, associative, and idempotent. Implemented by all 9 types.

DeltaCrdt

trait

Efficient delta sync. delta(&other) computes the minimal diff; apply_delta() applies it. GCounter, PNCounter, ORSet.

Quick Start

Basic Usage

Cargo.toml 
[dependencies]
crdt-kit = "0.3"
main.rs 
use crdt_kit::prelude::*;

// Two devices, working offline
let mut phone = GCounter::new("phone");
phone.increment();
phone.increment();

let mut laptop = GCounter::new("laptop");
laptop.increment();

// Reconnect — merge. Always converges.
phone.merge(&laptop);
assert_eq!(phone.value(), 3);

Delta Sync

delta_sync.rs 
use crdt_kit::prelude::*;

let mut sensor = GCounter::new("sensor-a");
sensor.increment_by(142); // 142 events detected

let mut gateway = GCounter::new("gateway");

// Delta: only send what the gateway doesn't have
let delta = sensor.delta(&gateway);
gateway.apply_delta(&delta);

assert_eq!(gateway.value(), 142); // up to date

With Persistence

persistence.rs 
use crdt_store::{CrdtDb, CrdtVersioned, SqliteStore};
use serde::{Serialize, Deserialize};

#[derive(Serialize, Deserialize)]
struct SensorReading { temperature: f32, humidity: f32 }

impl CrdtVersioned for SensorReading {
    const SCHEMA_VERSION: u8 = 1;
}

let store = SqliteStore::open("sensors.db").unwrap();
let mut db = CrdtDb::with_store(store);

// Save — wrapped in a version envelope
db.save("sensor-42", &SensorReading {
    temperature: 22.5, humidity: 65.0
}).unwrap();

// Load — auto-migrated if the schema changed
let reading: Option<SensorReading> = db.load("sensor-42").unwrap();

Schema Migrations

migration.rs 
use crdt_migrate::{crdt_schema, migration};

#[crdt_schema(version = 1, table = "sensors")]
#[derive(Serialize, Deserialize)]
struct SensorV1 { device_id: String, temperature: f32 }

#[crdt_schema(version = 2, table = "sensors")]
#[derive(Serialize, Deserialize)]
struct SensorV2 { device_id: String, temperature: f32, humidity: Option<f32> }

#[migration(from = 1, to = 2)]
fn add_humidity(old: SensorV1) -> SensorV2 {
    SensorV2 {
        device_id: old.device_id,
        temperature: old.temperature,
        humidity: None,
    }
}
// Loading a v1 record auto-migrates it to v2

Use Cases

IoT & Sensors

no_std core runs on ESP32, Raspberry Pi, bare metal. Delta sync minimizes bandwidth over LoRa/BLE. SQLite or redb for local persistence. Schema migrations handle OTA firmware updates seamlessly.

Mobile Apps

Offline-first architecture. Users edit data without network. When connectivity returns, all changes merge automatically. No conflict dialogs, no data loss.

Real-time Collaboration

TextCrdt for Google Docs-style editing. ORSet for shared collections. MVRegister to show concurrent edits. No central coordinator required.

Edge Computing

Deploy CRDTs at CDN edge nodes. Each node processes writes locally. Delta sync propagates changes between edges with minimal bandwidth. Pure-Rust redb backend — no C dependencies to cross-compile.

P2P Networks

No central server. Every peer is equal. Sync via any transport: WebSocket, WebRTC, Bluetooth, USB drive. Merge is commutative, associative, idempotent — order and duplicates don't matter.

WASM & Browser

Same CRDT logic runs in Rust backend and browser via WebAssembly. wasm-bindgen bindings for GCounter, PNCounter, LWWRegister, GSet, ORSet, TextCrdt. Ship one codebase everywhere.

Mathematical Guarantees

All CRDTs satisfy Strong Eventual Consistency (SEC). Verified by 268 tests across the workspace.

a ⊕ b = b ⊕ a

Commutativity

Order of sync doesn't matter. Device A syncing with B, or B with A, yields the same result.

(a ⊕ b) ⊕ c = a ⊕ (b ⊕ c)

Associativity

Group syncs however you want. Merge A+B first then C, or B+C first then A — same result.

a ⊕ a = a

Idempotency

Safe to retry. Duplicate messages, network replays, re-syncs — no double-counting, no corruption.
